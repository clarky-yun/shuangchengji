# 异地恋网站 PRD

## 1. 产品概述

### 1.1 产品名称

暂定：两座城之间

### 1.2 产品定位

一个专属于两个人的异地恋陪伴网站。它不是公开社交平台，而是一个私密、温柔、长期可维护的小空间，用来记录关系、同步日常、制造期待、保存回忆。

### 1.3 产品目标

- 让两个人随时感受到“对方就在生活里”。
- 把异地恋里最重要的信息集中展示：天气、距离、倒计时、在一起天数。
- 提供轻量互动：留言、分享日常、许愿、查看回忆。
- 支持长期沉淀：纪念日、见面记录、照片、共同愿望、重要片段。

### 1.4 目标用户

- 异地恋情侣，双方处于不同城市或不同国家。
- 希望拥有一个私密数字空间，而不是依赖朋友圈、聊天软件或备忘录。
- 喜欢记录纪念日、照片、心情和小目标的人。

## 2. 核心用户价值

### 2.1 情绪价值

- 打开网站即可看到彼此的城市、天气和下一次见面的倒计时，减少距离感。
- 留言和日常分享让对方即使忙碌也能感到被记挂。
- 回忆墙让关系有可视化的积累。

### 2.2 功能价值

- 自动计算在一起天数、见面倒计时、纪念日。
- 聚合双方城市天气。
- 支持留言、心愿、照片、回忆、日常动态。
- 支持私密访问，避免内容公开泄露。

## 3. 使用场景

### 3.1 每日打开

用户打开首页，看见：

- 今天是两个人在一起的第几天。
- 距离下次见面还有几天。
- 双方城市当前天气。
- 对方最近留下的一句话或一条日常。

### 3.2 想念对方时

用户进入留言区，写下一段话，可以选择：

- 普通留言。
- 悄悄话。
- 定时投递，例如“明早 8 点给 TA 看”。

### 3.3 分享生活

用户发布一条日常：

- 文本。
- 图片。
- 心情标签。
- 所在城市天气自动附带。

### 3.4 共同期待

用户创建或更新：

- 下次见面日期。
- 旅行计划。
- 共同心愿。
- 纪念日。

### 3.5 回顾过去

用户进入回忆页，按时间线查看：

- 第一次见面。
- 重要纪念日。
- 旅行照片。
- 已实现的心愿。
- 过去的留言和日常片段。

## 4. 功能范围

## 4.1 MVP 版本

MVP 目标：先做出一个可以真实使用的情侣私密主页。

### 4.1.1 首页仪表盘

展示内容：

- 两个人的昵称。
- 双方所在城市。
- 双方城市天气。
- 在一起天数。
- 距离下次见面倒计时。
- 最近一条留言。
- 最近一条日常。
- 最近一个共同心愿。

交互：

- 点击天气卡片查看更详细天气。
- 点击倒计时卡片编辑下次见面日期。
- 点击留言/日常卡片跳转到对应模块。

### 4.1.2 天气模块

功能：

- 展示双方城市当前天气。
- 展示温度、天气状况、空气湿度、风力。
- 支持手动配置城市。

MVP 简化：

- 先支持固定城市配置。
- 天气数据可以接入第三方 API，例如和风天气、OpenWeather、天气 API。

### 4.1.3 纪念日与倒计时

功能：

- 设置恋爱开始日期。
- 自动计算在一起天数。
- 设置下次见面日期。
- 自动计算距离见面的天数。
- 支持添加多个纪念日。

MVP 简化：

- 首页只显示恋爱天数和下次见面倒计时。
- 纪念日列表可放在独立页面。

### 4.1.4 留言板

功能：

- 发送文字留言。
- 查看留言列表。
- 留言显示发送人、时间、内容。
- 支持删除自己的留言。

后续增强：

- 悄悄话。
- 定时留言。
- 表情反应。
- 留言置顶。

### 4.1.5 日常分享

功能：

- 发布文字日常。
- 支持上传图片。
- 支持选择心情标签。
- 按时间倒序展示。

MVP 简化：

- 图片上传可先使用本地/云存储。
- 心情标签先内置固定选项：开心、想你、普通、疲惫、期待、委屈。

### 4.1.6 心愿清单

功能：

- 添加共同心愿。
- 设置心愿状态：未开始、进行中、已实现。
- 已实现心愿进入回忆页。

示例：

- 一起去海边看日出。
- 一起做一顿饭。
- 一起去对方城市生活一周。

### 4.1.7 回忆时间线

功能：

- 以时间线展示重要回忆。
- 支持标题、日期、文字、图片。
- 自动收录已完成心愿。

MVP 简化：

- 手动添加回忆。
- 后续再做从留言、日常、心愿中自动生成。

### 4.1.8 登录与权限

功能：

- 网站只允许两个人访问。
- 支持账号密码登录。
- 每条内容记录创建者。

MVP 简化：

- 两个固定账号。
- 不开放注册。

## 4.2 第二阶段功能

- 定时留言：指定时间后对方才能看到。
- 情绪日历：每天记录心情，形成月历。
- 共同相册：按地点、日期、主题整理照片。
- 见面计划：交通、住宿、行程、待办。
- 异地地图：显示两座城市和距离。
- 纪念日提醒：提前提醒生日、周年、见面日。
- 主题皮肤：根据季节、节日、纪念日更换视觉。
- 小纸条抽屉：随机抽取过去的一条留言。

## 4.3 第三阶段功能

- AI 回忆整理：自动从日常中总结“这个月我们一起经历了什么”。
- 语音留言。
- 视频片段收藏。
- 情侣问答游戏。
- 共同目标追踪，例如存钱、旅行、学习。
- 关系年度报告。

## 8. 关键业务规则

- 恋爱天数 = 当前日期 - startDate + 1。
- 见面倒计时 = nextMeetDate - 当前日期。
- 如果 nextMeetDate 是今天，展示“今天见面”。
- 如果 nextMeetDate 已过期，首页提示更新下一次见面日期。
- 留言按创建时间倒序排列。
- 日常按创建时间倒序排列。
- 已实现心愿可一键生成回忆。
- 所有用户内容默认仅两人可见。

## 9. 非功能需求

### 9.1 隐私

- 必须登录后访问。
- 不提供公开分享链接。
- 敏感配置如 API Key 不暴露在前端。

### 9.2 可用性

- 移动端优先，因为情侣更可能通过手机访问。
- 首页信息必须在首屏内基本可见。
- 发布留言和日常的入口要明显。

### 9.3 性能

- 首页首屏加载目标小于 2 秒。
- 天气 API 可缓存 10 到 30 分钟，避免频繁请求。
- 图片上传需限制大小，建议单图小于 5MB。

### 9.4 可维护性

- 功能按模块拆分：首页、天气、留言、日常、心愿、回忆、设置。
- 数据层和 UI 层分离。
- 天气服务封装成独立接口，方便更换 API。

## 10. 技术实现建议

### 10.1 推荐技术栈

适合个人项目、快速上线：

- 前端/全栈：Next.js
- UI：Tailwind CSS + shadcn/ui 或自定义轻量组件
- 数据库：Supabase Postgres 或 Firebase
- 认证：Supabase Auth / Clerk / NextAuth
- 图片存储：Supabase Storage / Cloudinary
- 天气 API：和风天气 / OpenWeather
- 部署：Vercel

### 10.2 更简单的首版方案

如果想先快速做出能看的版本：

- React + Vite
- 本地 JSON 或浏览器 localStorage 存数据
- 天气暂时用静态模拟数据
- 不做真实登录，只做访问密码

这个方案适合先看视觉和交互，但不适合长期真实使用。

### 10.3 推荐开发路线

第一步：静态可交互原型

- 首页、留言、日常、心愿、回忆、设置页面。
- 使用假数据。
- 完成整体视觉和交互。

第二步：接入数据存储

- 登录。
- 留言真实保存。
- 日常真实保存。
- 心愿和回忆真实保存。

第三步：接入外部服务

- 天气 API。
- 图片上传。
- 定时留言。
- 提醒功能。

## 12. 风格方向

### 12.1 视觉关键词

- 私密
- 温柔
- 清爽
- 有生活感
- 不幼稚
- 不像营销页

### 12.2 设计原则

- 首页像一个“关系状态小仪表盘”，不是宣传页。
- 尽量减少大段说明文字，让内容本身说话。
- 卡片数量控制，不要把页面做成信息堆叠。
- 色彩可以偏暖，但避免全粉色、全紫色。
- 移动端优先，桌面端增强。

### 12.3 建议组件

- 双城天气卡。
- 日期倒计时卡。
- 留言便签。
- 日常时间流。
- 心愿状态列表。
- 回忆时间线。
- 设置表单。

## 13. 风险与取舍

### 13.1 隐私风险

如果内容包含照片、私密留言，需要认真处理登录和访问权限。首版也应该至少有访问密码或登录机制。

### 13.2 功能膨胀

这个产品容易不断加入浪漫功能，但首版最好先保证：

- 每天会打开。
- 留言足够顺手。
- 日常能沉淀。
- 重要日期不会忘。

### 13.3 天气 API 稳定性

天气服务可能有额度限制或跨域问题，建议后端代理请求并做缓存。

### 13.4 图片存储成本

照片越来越多后会产生存储成本，首版需要限制图片大小和数量。

## 15. 一个可执行的首版范围

建议首版做成 6 个页面：

- 首页：天气、倒计时、天数、最近内容。
- 留言：两个人的留言板。
- 日常：文字和图片动态。
- 心愿：共同愿望清单。
- 回忆：时间线。
- 设置：城市、日期、昵称。

首版不做：

- 公开分享。
- 复杂社交功能。
- 多情侣账号系统。
- AI 自动总结。
- 视频和语音。

## 16. 下一步建议

如果进入开发，建议先做一个 Next.js 静态原型，确认视觉和交互之后再接数据库。这样可以最快判断这个网站是不是“打开后真的有感觉”，也能避免一开始就被登录、存储、API 配置拖慢。
